1. What Ansible Is (and Is Not)

Orientation

Ansible is an agentless automation engine for configuration, application deployment, and orchestrating other tools (like Terraform, Packer, Kubernetes, etc.). It’s state-driven but not declarative in the same way Terraform is.

Transport
Primarily SSH (Linux/Unix) and WinRM (Windows); no agents required on hosts.
Language
YAML + Jinja2 templates; modules written in Python (and other languages via plugins).
State
Desired state implied by tasks; Ansible modules are designed to be idempotent (run repeatedly with no change when already compliant).
Scope
Configuration, hardening, app deployment, operational workflows, API orchestration, compliance checks, ad-hoc tasks.
Terraform: build it Ansible: configure & run it Packer: image factory Kubernetes: schedule workloads
Architect’s angle: treat Ansible as your configuration and day-2 control plane. You use Terraform to create networks, VMs and managed services, then Ansible to harden, configure, deploy apps, and keep baselines in sync with your governance frameworks.

2. Terraform vs Ansible (Infra vs Config)

Design Boundary

Use Terraform and Ansible together on purpose, not by accident. The key is deciding which tool is the owner of truth for each layer.

Layer Use Terraform For Use Ansible For
Foundations
Accounts, projects, subscriptions, VCN/VNet/VPC		 Creating accounts/projects, VPC/VNet, subnets, gateways, routing, IAM roles/policies, base security groups. Rarely. Maybe to bootstrap APIs or initial configs where Terraform support is weak.
Compute & platform
VMs, databases, managed services		 Creating instances, databases, autoscaling groups, managed services endpoints. Hardening OS, installing agents, config files, patching, app runtimes, middleware.
Applications Wiring infra dependencies (LB targets, DNS records, secrets references). Deploying versions, updating config, restarting services, blue/green and rolling deploys.
Day-2 & Governance Drifting infra detection (via policy) and big, structural changes. Continuous hardening, CIS/STIG roles, periodic remediation, compliance evidence collection.

A common pattern is: Terraform creates hosts and core cloud services, then outputs inventory data; Ansible consumes that inventory to run hardening roles and deploy apps.

3. Inventory & Host Groups

Core Concept

Inventory tells Ansible what to talk to. Groups let you apply different behavior to different sets of hosts (env, tier, region, etc.).

Static inventory (INI/YAML)

[web] web01.example.com web02.example.com [db] db01.example.com [prod:children] web db [web:vars] ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/prod-web.pem
  • [group] headers define groups.
  • [group:children] lets you compose groups from other groups.
  • [group:vars] defines group-level variables (SSH user, etc.).

Dynamic inventory (cloud / CMDB)

Instead of hand-listing hosts, dynamic inventory scripts/plugins query a source of truth at runtime:

  • Cloud inventory (AWS, Azure, GCP, Oracle, VMware).
  • Service discovery / CMDB / IPAM.
  • Terraform state, if you choose to expose hosts that way.
Pattern: use Terraform outputs (or tags) to mark groups (role=web, env=prod), then configure dynamic inventory to build groups automatically: “all prod web servers in us-east-1”.

4. Playbooks & Plays

Core File

A playbook is the top-level YAML file; each play targets a set of hosts and runs tasks under a particular context.

--- - name: Harden and configure web servers hosts: web become: true vars: app_user: "app" app_port: 8080 pre_tasks: - name: Ensure apt cache is up to date (Debian/Ubuntu) ansible.builtin.apt: update_cache: true when: ansible_os_family == "Debian" roles: - role: cis_hardening - role: app_webserver post_tasks: - name: Verify app is listening ansible.builtin.wait_for: host: 0.0.0.0 port: "{{ app_port }}" timeout: 30
  • hosts – inventory group or pattern.
  • become – use privilege escalation (sudo).
  • pre_tasks, roles, post_tasks structure the play.

Multi-play playbook

Multiple plays in a single file target different groups in sequence:

--- - name: Configure database hosts: db become: true roles: - db_hardening - db_server - name: Configure web tier hosts: web become: true roles: - web_hardening - web_app

Think of a playbook as a workflow, and each play as a chapter in that workflow.

5. Tasks, Modules & Plugins

Action Engine

Tasks are individual steps in a play; modules are the building blocks that actually do the work (install packages, manage files, call cloud APIs, etc.).

Basic task anatomy

- name: Ensure nginx is installed ansible.builtin.package: name: nginx state: present - name: Ensure nginx is running ansible.builtin.service: name: nginx state: started enabled: true
  • name is human-readable and appears in logs.
  • Module name (e.g., ansible.builtin.service) identifies behavior.
  • Arguments are module-specific and generally idempotent.

Module categories

  • OS & system: packages, services, users, groups, files, lineinfile.
  • Cloud: EC2/VMs, load balancers, security groups, IAM, storage.
  • Network: routers, switches, firewalls (network_cli, httpapi).
  • Utilities: URI/HTTP calls, templates, command/shell (careful!), archives.

For governance, prefer dedicated modules over shell/command, so Ansible can track state and changes cleanly.

6. Variables, Facts & Templates (Jinja2)

Data & Rendering

Variables and templates let you drive environment-specific behavior from a common set of playbooks and roles.

Variable sources

  • Inventory vars (group_vars/, host_vars/).
  • Play vars (vars:, vars_files: in playbooks).
  • Role defaults (roles/role_name/defaults/main.yml).
  • Extra vars (-e on CLI, from CI/CD).
  • Facts (ansible_facts) gathered from hosts.

Simple template

# templates/nginx.conf.j2 server { listen {{ app_port }}; server_name {{ inventory_hostname }}; location / { proxy_pass http://127.0.0.1:{{ app_backend_port }}; } }

Using the template

- name: Render nginx config ansible.builtin.template: src: "nginx.conf.j2" dest: "/etc/nginx/conf.d/app.conf" owner: root group: root mode: "0644" notify: restart nginx

Jinja2 supports filters, conditionals, loops, and math. Use it to avoid copy-pasting near-identical configs across environments.

7. Roles, Collections & Folder Layout

Structure

Roles are reusable units of automation; collections are bundles of roles, modules, and plugins. This is where you codify your baselines and patterns.

Typical role layout

roles/ web_app/ defaults/ main.yml vars/ main.yml tasks/ main.yml handlers/ main.yml templates/ nginx.conf.j2 files/ meta/ main.yml
  • defaults/ – lowest-priority variables.
  • vars/ – higher-priority, often environment-specific.
  • tasks/ – main sequence of tasks for the role.
  • handlers/ – restart/reload actions triggered via notify.
  • templates/ and files/ – config and artifacts deployed by tasks.
  • meta/ – dependencies on other roles, metadata.

Collections

Collections bundle roles, modules, and plugins into versioned units that you can publish internally or consume from Ansible Galaxy.

  • Use collections to distribute cloud-specific logic (aws_baselines, azure_policies).
  • Pin collection versions in requirements.yml and your CI pipeline.
  • Mirror upstream collections into an internal artifact repository for air-gapped / regulated environments.
Your CCoE can ship “blessed” collections for STIG/CIS baselines, logging agents, and app patterns, so teams don’t reinvent everything.

8. Handlers, Conditionals, Loops, Blocks

Control Flow

Control flow features let you keep playbooks DRY, efficient, and safe even as complexity grows.

Handlers

- name: Install nginx ansible.builtin.package: name: nginx state: present notify: restart nginx handlers: - name: restart nginx ansible.builtin.service: name: nginx state: restarted
  • Handlers run at the end of a play, even if notified multiple times.
  • Good for restarts, reloads, and idempotent “expensive” actions.

Conditionals & loops

- name: Only run on Debian-family systems ansible.builtin.package: name: nginx state: present when: ansible_os_family == "Debian" - name: Create app users ansible.builtin.user: name: "{{ item }}" state: present loop: - app1 - app2

Blocks (block:, rescue:, always:) let you group tasks for error handling and cleanup; use them in complex orchestration.

9. Vault, Secrets & Sensitive Data

Security

Vault lets you encrypt variables and files so you can keep secrets in Git but still meet security requirements.

Encrypting secrets

# Encrypt a file ansible-vault encrypt group_vars/prod/secrets.yml # Edit an encrypted file ansible-vault edit group_vars/prod/secrets.yml # Re-key (rotate) vault password ansible-vault rekey group_vars/prod/secrets.yml
  • Use separate vault passwords per environment if needed.
  • Integrate vault passwords with a secrets manager / CI credential store.

Patterns

  • Keep secrets out of playbooks; reference variables instead.
  • Use vault-encrypted vars files and vars_files: in plays.
  • Limit who can decrypt which environments via RBAC and CI policies.
In regulated environments, log when vault credentials are used and by whom (e.g., via CI audit logs).

10. Execution Strategies & Performance

Scale

Execution strategies control how Ansible fans out work across hosts and how it behaves under load.

Forks & parallelism

The forks setting controls how many hosts Ansible works on in parallel. Tune per environment and job type.

  • Higher forks = faster runs, but more load on control node and targets.
  • Use lower forks for sensitive operations (DB changes, serial updates).

Serial & rolling

- hosts: web serial: 10 tasks: - name: Deploy app include_role: name: web_app

serial lets you roll out changes to a subset of hosts at a time (blue/green, canary, etc.).

11. AWX / Automation Controller Patterns

Platform

Centralized control (AWX / Automation Controller) turns ad-hoc playbooks into a governed automation platform.

Objects

  • Projects – Git-backed Ansible content.
  • Inventories & inventory sources – static/dynamic host lists.
  • Credentials – SSH keys, vault passwords, cloud keys.
  • Job templates & workflows – repeatable runs with parameters.

Governance patterns

  • RBAC by team, environment, and inventory.
  • Approval steps for risky jobs (DB changes, large rollouts).
  • Schedules for patching, baseline enforcement, compliance checks.
Treat Controller as the “front door” to automation for most users; only platform engineers run Ansible locally.

12. Cloud & Platform Modules (AWS · Azure · GCP · Oracle · VMware)

Cloud

Cloud modules let Ansible provision and manage resources directly. In most architectures, Terraform owns infra, but Ansible still touches cloud APIs for some day-2 workflows.

Where they fit

  • Emergency changes or remediation while Terraform is being updated.
  • Operational tweaks: security groups, firewall rules, instance tags.
  • Bulk operations (tagging, metadata updates, etc.).

Make sure governance requires that any “one-off” cloud changes via Ansible are quickly back-ported to Terraform.

Collections to standardize on

  • amazon.aws, community.aws
  • azure.azcollection
  • google.cloud
  • oracle.oci
  • vmware.vmware_rest / community.vmware

13. Golden Images, Hardening & Day-2 Ops

Baselines

Ansible is a natural fit for building golden images and continuously enforcing security baselines.

Golden image pipeline

  • Use Packer + Ansible provisioners to produce hardened images.
  • Tag images by baseline version, STIG profile, environment.
  • Publish to each cloud (AMI, managed image, custom image).

Day-2 enforcement

  • Scheduled jobs that re-apply hardening roles.
  • Playbooks that collect evidence (configs, logs, checksums).
  • Integration with SIEM and compliance tooling.

14. Compliance & Baselines with Ansible

Governance

Encode CIS, STIG, NIST and internal policies into Ansible roles so compliance is repeatable and testable.

Patterns

  • One role per baseline (e.g., rhel8_stig, cis_linux_level1).
  • Variables for profile level, environment exceptions, waivers.
  • Output evidence (JSON, text, reports) after each run.

Reporting

  • Use register: + debug / copy to save findings.
  • Ship results into SIEM / GRC tooling.
  • Track baseline drift over time as part of security posture.

15. Linting, Testing & CI/CD

Quality

Treat Ansible like application code: lint it, test it, and promote it through environments via CI/CD.

Linting & sanity checks

  • ansible-lint for playbook/role best practices.
  • YAML linters for syntax and style.
  • Custom rules for local policy (no shell unless justified, etc.).

Testing & pipelines

  • Molecule or similar frameworks for role testing.
  • CI pipelines that run lint + tests on every PR.
  • Promotion pipelines that target dev → test → prod inventories.
For highly regulated orgs, keep a full audit trail of which version of which role ran where, and with what parameters.

16. Command & Module Dictionary

Quick Reference

A practical dictionary of Ansible CLI commands and the most commonly used modules, grouped by what you use them for. This doesn’t list literally every module in the universe, but it covers the ones you keep reaching for in real architectures.

16.1 CLI Commands

Command What It Does Typical Use
ansible Runs ad-hoc tasks directly against inventory hosts. Quick one-off commands (e.g., ansible web -m ping) or simple changes without writing a playbook. Use sparingly in production; governance should constrain this.
ansible-playbook Executes YAML playbooks (the main way to use Ansible). Run your automation workflows, deploy apps, apply baselines. Supports --limit, --tags, --check, --diff, etc. Typical entrypoint for CI/CD jobs.
ansible-galaxy Manages roles and collections from Ansible Galaxy or private repos. ansible-galaxy collection install to pull cloud collections; publish and version your internal CCoE baselines, platform collections, and hardened roles.
ansible-vault Encrypts / decrypts files and vars (Vault). Securely store secrets in Git. Encrypt group_vars, edit them, rotate keys. Integrate with CI and secrets managers so humans don’t pass passwords around.
ansible-doc Shows documentation for modules, plugins and collections. ansible-doc ansible.builtin.service to see params, examples, and return values directly from the CLI. Encourage engineers to use this instead of random Googling.
ansible-config Views and manages Ansible configuration settings. Check which config file is active, see current values, understand how env vars/CLI flags override config. Good for debugging why a project behaves differently in CI vs local.
ansible-lint Lints playbooks, roles, and tasks for best practices. Wire into CI pipelines to catch bad patterns, deprecated modules, and style issues before merging. Use CCoE rules to encode your “house style”.
ansible-navigator TUI/CLI wrapper for running playbooks, validating content, exploring inventories. Modern way to interact with projects, especially in Automation Platform environments. Helpful for debugging inventories and exploring results interactively.
awx / tower-cli (legacy) CLI for AWX/Ansible Tower / Automation Controller. Manage projects, inventories, credentials, and job templates programmatically from scripts or CI/CD. Use where you want to treat Controller config as code.

For each CLI command, treat it as a building block you can script around. Governance comes from how you wrap these in pipelines and approval flows.

16.2 Core Built-in Modules (OS & Files)

Module Type / Category What It Does / When To Use
ansible.builtin.package Package management (generic) Abstract interface over OS package managers (apt, yum, dnf, zypper, etc.). Use when you want cross-distro playbooks that just say “install this package”.
ansible.builtin.apt Package management (Debian/Ubuntu) Directly use apt features: caches, state, upgrades; good when you need apt-specific options or fine-grained control on Debian-family systems.
ansible.builtin.yum / dnf Package management (RHEL/CentOS/Fedora) Manage RPM packages; pin versions; handle repos for RHEL-based distros. Use for long-lived enterprise fleets.
ansible.builtin.service Service management Start/stop/restart and enable services. Core for keeping daemons running (nginx, sshd, app services, etc.). Almost every role touches this at some point.
ansible.builtin.user User accounts Create, modify, and remove system users; manage shells, home dirs, passwords, groups. Map to least-privilege controls and app identities.
ansible.builtin.group Groups Manage local groups; often used with user to enforce local RBAC patterns, especially on bastion hosts or app nodes.
ansible.builtin.file Filesystem / attributes Ensure files/directories/symlinks exist or are absent; set ownership, permissions, SELinux context. Core for CIS/STIG enforcement.
ansible.builtin.copy File content Copy static files from the control node to managed nodes. Use for simple config files that don’t need templating.
ansible.builtin.template File content (templated) Render Jinja2 templates using vars/facts. Use whenever config depends on environment, host, or secrets; core for multi-env patterns.
ansible.builtin.lineinfile Config tweaking Ensure a specific line is present/absent or matches a regex in a file. Good for minor config adjustments where full templating isn’t worth it (but don’t abuse it).
ansible.builtin.blockinfile Config blocks Manage a multi-line block of text in a file; great for inserting managed sections in config files and leaving the rest alone. Helpful with vendor-managed configs.
ansible.builtin.shell / command Raw command execution Run arbitrary commands. Use only when there is no dedicated module, and be careful with idempotence and quoting (prefer command when you don’t need a shell).
ansible.builtin.debug Logging / output Print messages or variable values for debugging. Also useful to emit evidence into logs for governance and audits.
ansible.builtin.stat File/dir facts Gather info (exists, size, hashes, permissions) about paths. Combine with when: for conditional logic and idempotent workflows.
ansible.builtin.unarchive Archives Unpack tar/zip/etc. archives locally or remotely, with optional remote_source; useful for installing apps from tarballs or delivering artifacts.
ansible.builtin.get_url HTTP/HTTPS download Download files from URLs to the target; good for fetching artifacts from web servers, object storage, or artifact repos.
ansible.builtin.git Source code / repos Clone or update Git repos on managed hosts; version-pin dependencies or app code. Common in app deployment roles.

16.3 Control & Flow Modules

Module Type What It Does / When To Use
ansible.builtin.include_role / import_role Role inclusion Dynamically (include_role) or statically (import_role) include roles inside tasks; allows you to parameterize role use or conditionally run roles.
ansible.builtin.include_tasks / import_tasks Task inclusion Split large roles/playbooks into multiple files; reuse sequences of tasks across plays with conditions. Helps keep repos readable.
ansible.builtin.set_fact Dynamic variables Create or update host-level variables during a run, based on calculations or lookup results. Use sparingly and document clearly, as it can hide logic.
ansible.builtin.assert Validation Fail early if assumptions aren’t met (e.g., required vars missing, conditions not satisfied); great for governance and guardrails in shared roles.
ansible.builtin.wait_for Readiness Wait for ports, files, or conditions (e.g., service listening on port 443) before moving on; helps orchestrate multi-tier rollouts safely.
ansible.builtin.uri HTTP/REST Call HTTP(S) APIs; integrate with cloud APIs, CMDBs, or governance systems when a dedicated module doesn’t exist. Also useful for talking to internal platforms.
ansible.builtin.fetch File collection Copy files from remote hosts back to the control node (logs, configs, evidence) for troubleshooting and audits.
ansible.posix.synchronize Efficient sync rsync wrapper; efficient for syncing directories between control node and managed hosts (or between hosts) at scale.

16.4 Cloud & Governance-Relevant Modules (Examples)

Module (Collection) What It Controls Why It Matters for Governance
amazon.aws.ec2_instance Provision and manage EC2 instances. Used for patching, emergency remediation, or maintenance actions when you must touch instances directly (though Terraform typically owns infra).
amazon.aws.ec2_group Security groups (AWS). Tighten or audit security group rules in response to findings; can be used for emergency changes while Terraform is updated later. Ensure drift is reconciled.
azure.azcollection.azure_rm_virtualmachine Azure VMs lifecycle. Similar use to EC2: patching and operational activities around Terraform-managed VMs. Often combined with extensions and agents.
google.cloud.gce_instance GCE compute instances. GCP fleet operations and patching for hosts in P2/P3 environments. Use tags/labels for grouping and targeting.
oracle.oci.oci_compute_instance OCI compute instances. Hybrid/OCI workloads; apply hardening roles and agent installs programmatically in Oracle environments.
community.general.timezone System timezone Enforce consistent time settings across fleets for log correlation and incident response; important in regulated environments.
community.general.seboolean SELinux booleans Align SELinux settings with STIG/CIS controls; important in high-assurance environments where SELinux is enforced.
ansible.posix.firewalld firewalld configuration Enforce host-level firewalls; good for defense-in-depth beyond cloud security groups, especially for East/West controls.

Treat this dictionary as a living thing: extend it whenever you standardize on new modules or patterns. The rule of thumb is: if engineers keep Googling “how do we do X in Ansible?”, it probably deserves a row here.
