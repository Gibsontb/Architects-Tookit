1. What “Single Source of Truth” Means

Orientation

This is not just “a wiki” or “a repo.” It’s a model for where the truth lives: what is authoritative, what is derived, and what is just a working scratchpad.

Goal
Everyone—from architect to auditor—knows exactly where the real, current answer lives for any architecture, policy, or automation question.
Principle
One authoritative location per artifact type. Mirrors and copies are allowed but never the truth; they always reference the canonical source.
Audience
CCoE, architects, engineers, security, compliance, operations, and program managers for migrations and day-2 operations.
Authoritative Traceable Versioned Auditable Discoverable
Architect’s rule: if you can’t answer “where is the truth for this?” in under 5 seconds, you don’t have a single source of truth yet. Use this page to fix that.

2. Scope & Boundaries

Scope

Define exactly what this registry covers so it doesn’t try to own everything, and so people know what still lives elsewhere.

2.1 In Scope

  • Architecture standards, patterns, and reference diagrams.
  • Cloud landing zone designs and environment mappings.
  • Governance mappings (regulation → framework → control → implementation).
  • Terraform & Ansible module catalogs and ownership.
  • Runbooks, playbooks, and operating procedures.
  • Key decisions (ADR/SBR style) and rationale.

2.2 Out of Scope

  • Transient tickets and chats (Jira, email, Slack) — link them, don’t store them.
  • Raw logs and metrics (these live in observability tools).
  • Highly local, team-only scratch docs (keep them, but they’re not canonical).

The registry points to where those things live and captures their impact, but does not try to duplicate them.

4. Object Types & Examples

Objects

“Single source of truth” is about objects: each object has a type, an ID, an owner, and a canonical home.

4.1 Core Object Types

  • Standard – a normative rule (e.g., “all P3 data must be encrypted at rest”).
  • Pattern – a reusable architecture or implementation blueprint.
  • Module – Terraform/Ansible module or role with versioning.
  • Runbook – step-by-step operations procedure.
  • Decision Record – why a choice was made, with date and approvers.

4.2 Example IDs

  • STD-SEC-001 – Encryption at Rest for P3 Data.
  • PAT-LZ-COURT-STD – Standard Court Landing Zone Pattern.
  • MOD-TF-NET-TRANSIT-01 – Terraform Transit Network Module.
  • RUN-INC-004 – Incident Response: Account Compromise.
  • ADR-ARCH-017 – Decision: Primary Cloud for Court Case Mgmt.
Prefix by type Numeric sequence Optional domain suffix

6. Identifiers & Versioning

Identity

Every object needs a stable identifier and a way to talk about versions in emails, tickets, and audits.

6.1 ID Scheme

  • Prefix by object type (STD, PAT, MOD, RUN, ADR, etc.).
  • Optional domain (e.g., SEC, NET, LZ, APP, DATA).
  • Numeric sequence: 3–4 digits is usually enough.

Example: PAT-LZ-COURT-STD-002 → second standard court landing zone pattern.

6.2 Versioning

  • Semantic versioning for living artifacts:
    • MAJOR – breaking changes to behavior or guarantees.
    • MINOR – new capabilities, backwards compatible.
    • PATCH – documentation fixes or risk clarifications.
  • Tag repos and docs with both ID and version:
    • Example: PAT-LZ-COURT-STD-002 v1.3.0.

7. Create / Change / Approve Workflows

Process

The registry is only as good as the workflows behind it. This section captures how artifacts are created, reviewed, and published as truth.

7.1 Create / Update Flow

  1. Proposal created in working branch or draft doc.
  2. Reviewers assigned (architects, security, ops, business owner).
  3. Comments resolved; ADR/SBR raised if it’s a big directional change.
  4. On approval, artifact gets a new version and updates this registry.
  5. Communication to impacted teams and updates to training/runbooks.

7.2 Approvals

  • Define who is required vs optional for each type:
    • Standards: CCoE + security + business sponsor.
    • Patterns: CCoE + platform + security (for P3 patterns).
    • Modules: module owner + platform + security (if it touches P3).
  • Approval logged as part of the artifact’s metadata:
    • Approver names, date, and any conditions or exceptions.

8. Lifecycle & Deprecation

Lifecycle

Truth changes. Old standards and patterns need a clear way to be deprecated so nobody is stuck implementing dead ideas.

8.1 States

  • Draft – being written, not yet authoritative.
  • Proposed – under review with identified approvers.
  • Approved – canonical and ready for implementation.
  • Deprecated – still visible, but replaced by something else.
  • Retired – archived; no longer used except for audit history.

8.2 Deprecation Rules

  • New patterns and modules reference which older items they supersede.
  • Deprecated items remain discoverable with a clear pointer:
    • “Replaced by PAT-LZ-COURT-STD-003 v2.0.0.”
  • Retirement after a defined sunset period (e.g., 12–24 months) when no active systems still depend on them.

9. Integration with CCoE, Terraform, Ansible & Governance

Integration

This registry is the glue between your org model (CCoE), your code (Terraform & Ansible), and your governance (regulations & controls).

9.1 CCoE & Landing Zones

  • Each landing zone pattern (ccoe.html) has a matching entry here.
  • For each pattern, list:
    • Pattern ID and version.
    • Linked Terraform modules and Ansible roles.
    • Associated standards and controls.

9.2 IaC & Governance

  • For key controls in governances.html, map:
    • Regulation / framework control.
    • Terraform modules implementing it.
    • Ansible roles enforcing it on hosts.
  • Ensure module and role IDs appear both here and in their own catalogs.

11. Implementation Checklist

Execution

Use this to track bringing your “single source of truth” from concept to reality. Copy and paste into your task tracker if needed.

  • Agree on domains and object types (standards, patterns, modules, runbooks, decisions).
  • Define ID scheme and versioning rules.
  • Choose systems of record for each domain (repos, doc stores, catalogs).
  • Wire this page into index.html and other workspaces.
  • Register existing key artifacts and assign IDs.
  • Define workflow for create/change/approve per object type.
  • Document lifecycle states and deprecation rules.
  • Map governance controls to IaC modules and roles.
  • Validate with at least one audit or tabletop exercise.
  • Review yearly (or quarterly) and update as the org matures.

12. Roadmap & Hardening

Roadmap

Implement the single source of truth in phases so it doesn’t stall under its own weight.

Phase Focus Key Actions
Phase 0 – Baseline Define domains, IDs, and where things live. Stand up this page, connect it to existing Terraform/Ansible/Governance/CCoE workspaces.
Phase 1 – Registration Register top 20–50 critical artifacts. Assign IDs, versions, owners, and states; clean up obvious duplicates.
Phase 2 – Workflow Enforce create/change/approve flows. Update contribution guidelines; add registry steps to change management and reviews.
Phase 3 – Automation & Audit Automate lookups and reporting. Expose registry as data (JSON/CSV); generate reports for courts, execs, and auditors.

Over time, the registry becomes just “how we work,” not an extra task.
