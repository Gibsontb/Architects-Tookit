1. What “Single Source of Truth” Means

Orientation

This is not just “a wiki” or “a repo.” It is the central, authoritative repository for all material the courts rely on — structured data, documents, evidence, and daily work artifacts like calendars, schedules, and shared files — across Supreme Court, FISA, appellate, and hundreds of other courts.

Goal
Everyone—from clerk to architect to auditor—knows exactly where the real, current version of any case file, document, photograph, recording, warrant, dataset, calendar, schedule, or work artifact lives, with clear tags and security classification.
Principle
One authoritative location per object (case file, evidence item, schedule, standard, pattern, runbook). Mirrors and copies are allowed but never the truth; they always reference the canonical source with IDs and tags.
Audience
CCoE, architects, engineers, security, clerks, judges, operations, and program managers for migrations and day-2 operations across all courts.
Authoritative Centralized Tagged Security-aware Court-wide Auditable
Architect’s rule: if you can’t answer “where is the truth for this case, warrant, recording, or court schedule?” in under 5 seconds, you don’t have a single source of truth yet. Use this design to point everything to one central location.

2. Scope & Boundaries

Scope

Define exactly what this registry and repository cover so it owns “court truth” (all data + all docs + evidence + daily work + technical design), without trying to duplicate transient systems.

2.1 In Scope

  • Core court data sets (case records, parties, dockets, calendars, dispositions).
  • Court documents and files (orders, motions, filings, briefs, warrants, subpoenas, PDFs, scans, attachments).
  • Evidence assets: photographs, bodycam and audio recordings, videos, forensic exports, large media files and external file references.
  • Personal and sensitive data attached to cases and operations (where allowed by law).
  • Daily work artifacts produced by court personnel:
    • Judicial and courtroom calendars & schedules.
    • Staff task lists, shift schedules, duty rosters.
    • Shared drives, Teams/SharePoint/OneDrive/Google Drive artifacts tied to court work.
    • Templates and forms used for daily operations.
  • Collaboration platform structures and policies: Office 365 / M365, Google Workspace (mailboxes, groups, teams, sites, channels).
  • Architecture standards, patterns, and reference diagrams.
  • Governance mappings (regulation → framework → control → implementation).
  • Terraform & Ansible module catalogs and ownership.
  • Runbooks, playbooks, and operating procedures.
  • Key decisions (ADR/SBR style) and rationale.

2.2 Out of Scope

  • Transient tickets and chats (Jira, email, Slack/Teams chat) — link them, don’t store them here.
  • Raw logs and metrics (those live in observability tools).
  • Local scratch docs or personal notes (they can reference registry objects).

The single source of truth points to where those things live and captures their impact, but does not try to be an inbox, chat history, or log store.

4. Object Types & Examples

Objects

“Single source of truth” is about objects: case files, evidence items, documents, work artifacts, datasets, standards, and modules. Each object has a type, an ID, tags, and a canonical home.

4.1 Core Object Types

  • Case File – canonical record for a court case, with ID and tags.
  • Document – individual filing/order (PDF, scan, brief, warrant, etc.).
  • Evidence Item – photographs, recordings, media, or other artifacts tied to a case.
  • Dataset – curated court dataset (e.g. “Daily Docket Extract”).
  • Work Artifact – calendar entry, schedule, roster, or shared file tied to court operations.
  • Standard – normative rule (e.g., “all P3 data must be encrypted at rest”).
  • Pattern – reusable architecture or implementation blueprint.
  • Module – Terraform/Ansible module or role with versioning.
  • Runbook – step-by-step operations procedure.
  • Decision Record – why a choice was made, with date and approvers.

4.2 Example IDs & Tags

  • CASE-2025-CR-001234 – Case file with tags: [court=SUP], [type=Criminal], [sensitivity=P3].
  • DOC-CASE-2025-CR-001234-ORD-01 – Order document linked to the case.
  • EVID-CASE-2025-CR-001234-PHOTO-01 – Photograph evidence linked to the case.
  • EVID-CASE-2025-CR-001234-AUDIO-01 – Audio recording linked to the case.
  • WORK-SUP-COURTROOM-01-2025-03-01-AM-DOCKET – Courtroom schedule for a day/slot.
  • DATA-DOCKET-DAILY – Docket data mart export, tagged for reporting only.
  • STD-SEC-001 – Encryption at Rest for P3 Data.
  • PAT-LZ-COURT-STD – Standard Court Landing Zone Pattern.
  • MOD-TF-NET-TRANSIT-01 – Terraform Transit Network Module.
Prefix by type Case / doc / evidence / work IDs Tags for court, type, sensitivity Stable URLs / URIs

6. Identifiers, Tagging & Versioning

Identity

Every object needs a stable identifier, tags (for search and security), and a way to talk about versions in emails, tickets, and audits.

6.1 ID Scheme

  • Prefix by object type (CASE, DOC, EVID, DATA, WORK, STD, PAT, MOD, RUN, ADR, etc.).
  • Include jurisdiction / court and optional domain (e.g., SUP, FISA, APP, CR).
  • Numeric sequence or structured timestamp for work artifacts (calendars/schedules).

Example: WORK-SUP-COURTROOM-A-2025-09-10-AM → canonical schedule for Courtroom A’s morning session on that date, tied to case entries and staff.

6.2 Tagging & Versioning

  • Standard tag sets:
    • Classification: P1, P2, P3, CJIS, Juvenile, Sealed, FISA.
    • Domain: Criminal, Civil, Family, Traffic, FISA, Appellate, Admin.
    • Lifecycle: Open, Scheduled, Closed, Archived.
    • Retention: standard, extended, legal hold.
    • Work type: Courtroom schedule, staff roster, meeting, training, etc.
  • Semantic versioning for living artifacts (standards, patterns, modules, runbooks):
    • MAJOR – breaking changes to behavior or guarantees.
    • MINOR – new capabilities, backwards compatible.
    • PATCH – documentation fixes or risk clarifications.
  • Document/evidence/work versions can be either:
    • Immutable “record per version” with tags (DOC-...-v3, EVID-...-v2, WORK-...-v5), or
    • Single ID with a version history managed in the DMS/evidence/collaboration system.

7. Create / Change / Approve Workflows

Process

The registry is only as good as the workflows behind it. This captures how case records, documents, evidence, work artifacts, datasets, and standards are created, tagged, secured, reviewed, and published as truth.

7.1 Data, Documents, Evidence & Work Flow

  1. New case/doc/evidence/work item is created in the source system (CMS, DMS, evidence store, M365/Workspace).
  2. It is assigned a canonical ID and stored in the central data/doc/evidence/work platform.
  3. Tags are applied (court, type, sensitivity, retention, lifecycle, work type).
  4. The registry entry is created/updated to point at the canonical object.
  5. Security and access groups are evaluated based on tags and legal constraints.
  6. Scheduled items (calendars, rosters) are linked back to cases and rooms where appropriate.

7.2 Standards & Technical Artifacts

  • Proposal created in working branch or draft doc.
  • Reviewers assigned (architects, security, ops, court business owner).
  • Comments resolved; ADR/SBR raised if it’s a big directional change.
  • On approval, artifact gets a new version and updates this registry.
  • Communication to impacted teams and updates to training/runbooks.

8. Lifecycle & Deprecation

Lifecycle

Truth changes. Cases close, documents and warrants are superseded, evidence is reprocessed, schedules are updated, and standards evolve. Lifecycle and tags keep the central repository clean without losing history.

8.1 States

  • Draft – being written, not yet authoritative (for standards/patterns).
  • Active – current and authoritative (case/doc/evidence/work or standard in force).
  • Superseded – replaced by a newer version, still visible for history.
  • Scheduled – work items like calendars/rosters that are upcoming.
  • Completed / Closed / Archived – case closed, hearing held, schedule completed, retained per policy.
  • Retired – no longer used except for audit history.

8.2 Deprecation Rules

  • New standards and patterns reference which older items they supersede.
  • New documents/orders reference the prior order they supersede when relevant.
  • Evidence reprocessing keeps original artifacts, but marks them as superseded by the new version.
  • Schedule changes keep prior versions visible for audit (what was on the calendar at a given time).
  • Deprecated or superseded items remain discoverable with a clear pointer:
    • “Replaced by DOC-CASE-...-ORD-02 v2.0.0.”
  • Retirement after a defined sunset period when no active systems depend on them.

9. Integration with CCoE, Terraform, Ansible, Governance & Collaboration

Integration

This registry is the glue between your central court data/doc/evidence/work repository, your CCoE architecture, your code (Terraform & Ansible), your governance, and your collaboration suites (Office 365 / M365, Google Workspace).

9.1 CCoE & Landing Zones

  • Each landing zone pattern (ccoe.html) defines where central court data, documents, evidence, and collaboration systems physically live (regions, buckets, DBs, tenants).
  • For each pattern, list:
    • Pattern ID and version.
    • Linked Terraform modules and Ansible roles.
    • Associated standards and controls.

9.2 IaC, Governance, Security & Collaboration

  • For key controls in governances.html, map:
    • Regulation / framework control.
    • Terraform modules implementing it (encryption, retention, tagging, tenant/O365/Workspace settings).
    • Ansible roles enforcing it on hosts and data/evidence/collaboration services.
  • Ensure data/doc/evidence/work tags (sensitivity, classification, retention, court, work type) drive IAM policies and group membership in Cloud Identity / Google Workspace / Entra ID / O365.
  • Use tags to drive default retention and legal hold on collaboration data (mailboxes, Teams/SharePoint, Google Drive/Docs).

11. Implementation Checklist

Execution

Use this to track bringing your central, tagged, secure “single source of truth” from concept to reality. Copy and paste into your task tracker if needed.

  • Agree on domains and object types (cases, docs, evidence, work, datasets, standards, patterns, modules, runbooks, decisions).
  • Define ID scheme and versioning rules for case/doc/evidence/work and technical artifacts.
  • Design and lock down the tagging model (classification, domain, lifecycle, retention, FISA/CJIS/juvenile, work types, etc.).
  • Choose systems of record for each domain (data lake/warehouse, DMS, evidence system, collaboration suite, repos, catalogs).
  • Wire this page into index.html and other workspaces.
  • Register existing key artifacts and assign IDs and tags.
  • Define workflow for create/change/approve per object type.
  • Enforce security & access rules based on tags and group membership.
  • Map governance controls to IaC modules, roles, and tagged data/doc/evidence/work domains.
  • Validate with at least one audit or tabletop exercise focused on court records, evidence, and daily work artifacts.
  • Review yearly (or quarterly) and update as the courts and program mature.

12. Roadmap & Hardening

Roadmap

Implement the single source of truth in phases so it becomes the default way the courts operate, not just another project.

Phase Focus Key Actions
Phase 0 – Baseline Define domains, IDs, tags, and where things live (data/doc/evidence/work). Stand up this page, connect it to existing case/evidence systems, collaboration suites, data platforms, Terraform/Ansible/Governance/CCoE workspaces.
Phase 1 – Registration Register top 20–50 critical datasets, doc collections, evidence collections, and operational calendars. Assign IDs, tags, versions, owners, and states; clean up obvious duplicates and shadow systems.
Phase 2 – Workflow Enforce create/change/approve flows using IDs + tags. Update contribution guidelines; add registry steps to change management and court operations.
Phase 3 – Automation & Audit Automate lookups, security decisions, and reporting from tags. Expose registry as data (JSON/CSV); generate reports for courts, execs, and auditors across all 300+ courts.

Over time, the single source of truth becomes the only place people look for court data, documents, evidence, and work artifacts—and everything else is just a view on top of it.
